# 引き継ぎメモ: モビリティ・カルテ(出口C)の追加要望

- 作成日: 2026-07-15
- 作成元: 別のClaude Codeセッション(アイデア検討担当)。本文書は「何を作りたいか」の要件メモであり、
  実装計画(マイルストーン分割・完成条件)は**このセッション(norishiro担当)が
  plan_gap_map.md の流儀で作成する**こと。

## 1. 何を作るか(一言で)

既存の空白マップは「地域の診断」。これを「**個人の診断書**」に読み替える入口を1枚追加する。

> 住所を入れる(または地図をクリックする)と、
> 「あなたの家の移動健康診断書」が1枚のカードで表示される。

## 2. なぜ作るか(作品ストーリー上の狙い)

- 現状の構成は「第1部=大活字ペア時刻表」「第2部=空白マップ」で、審査員には
  2作品の同居に見えるリスクがある。
- カルテを入口にすると「**診断(カルテ)→処方箋(あなた専用の時刻表)**」という
  一本のストーリーになる: 住所を入れる → 診断書が出る →
  「あなたの家から病院への時刻表はこちら」と第1部・出口Bへ接続する。
- 分析はもともと車なし前提なので、計算を変えずに
  「**免許返納シミュレーション**(車を手放したらこうなる)」という枠付けができる。
  少子高齢化・交通空白という2026年の重点課題に個人視点で刺さる。

## 3. 診断書カードに載せる内容(案)

すべて access_mesh.csv 等の**計算済みデータの読み替え**で賄えるはず。新規計算は不要。

- 最寄りバス停(名前・徒歩○分)
- 最寄り病院まで公共交通で○分 / スーパーまで○分
- 通院可能性(指標②): 可否と拘束時間
- 市平均(または地区平均)との比較(「あなたの地区は山形市平均より○分長い」)
- 総合評価(例: A〜Eの5段階。しきい値はconfig化し、plan_gap_map.md §7の空白判定と整合させる)
- 第1部/出口Bへの導線(「この家からの時刻表を見る」)

## 4. 制約・注意(検討セッションで合意済みの方針)

1. **プライバシー**: 住所→緯度経度の変換はブラウザ内で完結させ、住所をサーバーに送らない・
   記録しない。国土地理院ジオコーディングAPI(https://msearch.gsi.go.jp/address-search/AddressSearch 、
   無料・キー不要)をクライアントサイドから呼ぶ想定。GitHub Pages構成なら自然に満たせる。
   READMEにも明記する(審査観点)。
2. **スコープ管理**: まず「住所→診断カード」だけを完成させる。
   以下は**オプション**とし、コアが動くまで着手しない:
   - 将来人口の重ね合わせ(国土数値情報の500mメッシュ別将来推計人口で「10年後のあなたの地区」)
   - 診断書のPDF/印刷対応
3. 緯度経度→メッシュコード変換は meshcode.py に実装済みのはず。Web側で使うなら
   JavaScriptへの移植が必要になる点に注意(export_web_data.py で事前埋め込みする手もある)。
4. 「診断書」「カルテ」は比喩。医療的な診断を連想させる断定表現は避ける。

## 5. このセッションへの依頼

(または docs/plan_karte.md として新規作成)してほしい。
計画ができたら、これまで通りマイルストーン単位で開発者本人が発注する。
