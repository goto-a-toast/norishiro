## 1. プロジェクトの目的と「3つの顔」

**課題**: 山形市・上山市では、公共交通の空白地帯に15,418人(人口の5.6%)が住み、
その高齢化率は43.2%と2市平均30.3%より12.9ポイント高い。
「移動に困る場所ほど、移動が困難な人が住んでいる」——これを可視化し、
高齢者本人・家族・行政のそれぞれが使える形で解決策を提供する。

**作品の構成 = 1つの自作計算エンジンから3つの顔を供給する**:

| 顔 | 対象者 | 実体 | 状態 |
|---|---|---|---|
| ① かんたんモードWeb | 高齢者本人・窓口支援者 | `webapp/`(3タップで時刻表。静的サイト) | **進行中**(F4は実装完了。実機検収・第三者テストのみ残) |
| ② 大活字・往復ペア時刻表PDF | 窓口で配る紙 | `make_pair_timetable.py` | **完成**(QR欄追加=F7のみ残) |
| ③ 交通空白マップ | 政策判断・審査員 | `gap_map/` → `output/gap_map.html` | **完成・凍結済み** |

データの流れ:

```
【Python側 = データ供給工場(手元で実行)】
 9フィードGTFS ─→ transit_core(凍結済みRAPTOR) ─→ export_web_data.py(F3・完成)
                                                        │ 事前計算JSON一式
                                                        ▼
【Web側 = 静的サイト(GitHub Pages。サーバー計算なし)】
 webapp/ … 玄関 + かんたんモード + しっかりモード + 空白マップ + data/*.json
```

## 3. 主要な設計判断とその理由

引き継ぎ後にこれらを覆す場合は、理由が現在も成立しないことを確認してから。

### 3.1 r5py不採用 → 簡易RAPTOR自作(plan_gap_map.md §4)

- r5py はJava製エンジンR5のラッパー。導入障壁が3つ重なる:
  JDK21(このMacはmacOS 12でHomebrew不調、手動.pkg導入になる)+
  OSM PBF切り出し(osmium等の別ツール)+ jpypeトラブルのリスク
- 規模的に不要: 県全域(推定1万停留所)でもRAPTORで1出発点0.1秒未満
- **検証可能性が決め手**: 全部Pythonなので初心者でも追え、第1部のPDF
  (山形駅前→県立中央病院 乗車約27分)と直接突き合わせられる
- 作品性: 「エンジンを自作し3用途に使い回した」と言える
- 逃げ道: `transit_core.py` の入出力境界(§3.3)を守っているので、
  r5pyへの差し替えは呼び出し側1関数の置換で済む

### 3.2 エンジンと地理データの分離(plan_gap_map.md §1。最重要ルール)

`transit_core.py` は「バス停」と「時刻」だけを知る。メッシュ・施設・人口・GTFSの
読み方は一切持ち込まない。

- 入力: 「出発できるバス停と、そこに立てる時刻」の辞書
- 出力: 「全バス停への最早到着時刻」+経路復元用の記録
- 空白マップは「メッシュ中心から徒歩圏のバス停」を、経路検索は「指定バス停」を
  入力にするだけの違い。**この境界のおかげで同じエンジンが3つの顔に使える**
- 付随ルール: 全フィードのstop_id/trip_idに「フィード名:」接頭辞を付ける
  (フィード間ID衝突の防止。build_network.py が実施)

### 3.3 静的サイト方式(plan_final_sprint.md §1)

- **Webアプリは計算しない**。Python側で事前計算したJSONを読んで表示するだけ。
  「今日の日付→ダイヤ種別」の判定すら meta.json の対応表を引くだけにする
  (祝日・お盆の特殊ダイヤ判定をJSに書かない。バグの温床を根絶する設計)
- 素のHTML/CSS/JS。フレームワーク・ビルドツール・CDNなし。
  `webapp/` をそのままアップロードすれば動く
- 理由: 運用費ゼロ(GitHub Pages)、自治体がフォークして再生成できる、
  初心者開発者が自力で保守できる
- デザインは第1部PDFの紙面を継承(ごぜん/ひる/ごご/よるの段、行き=青・帰り=橙、
  BIZ UDPゴシック)。「紙とWebで同じ見た目」が作品の一貫性

### 3.4 学区ハイブリッド方式(plan_final_sprint.md §3)

かんたんモードの「地区」= 小学校区(「◯◯小学校の学区」は高齢者に通じる地域の共通語)。

- 山形市: 国土数値情報 **A27**(小学校区ポリゴン・平成28年版)を使用。36学区あり
- **問題**: A27山形県版は30市町村のみ収録で**上山市が未収録**(2026-07-06確認)
- **対応**: 上山市は国土数値情報 **P29**(学校・点データ、全国完全収録)の小学校位置から
  「各メッシュを最寄りの小学校に割り当てる」近似で学区相当の地区を作る。
  OSMは上山市の小学校が1校しか登録されておらず不採用
- 近似であることは応募資料の限界の章に明記する
- 地区の代表点 = その学区内で人口最大の500mメッシュの中心(target_meshes.csv から機械的に決定)
- 学区名が住民感覚と合わないリスクは、districts.json に人間が上書きできる「表示名」列で吸収

### 3.5 その他の確定済み判断

| 判断 | 内容 | 理由 |
|---|---|---|
| メッシュ粒度 | 500m(標準地域メッシュ4次) | e-Stat人口データとマス目が一致 |
| 事前計算の範囲 | 出発地区=山形市+上山市(41地区。F1で確定)、路線網=9フィード | 「狭い」のではなく出発点を絞るだけ。市境をまたぐ便も正しく出る。ボトルネックは計算でなく人手の品質作業 |
| 対象日 | 2026-06-10(水・平日・非祝日) | フィード有効期間内でダイヤ例外の影響なし |
| パラメータ | すべて `gap_map/config.py` に集約 | 「しきい値を変えたら?」に1コマンドで答えるため |
| 施設 | 病院=国土数値情報P04(病院コード1のみ)、スーパー=OSM(supermarketのみ) | 診療所・コンビニは論点が変わるためconfig切替で第1弾は除外 |
| 乗換1回経路の紙面(PDF)化 | **やらない**(展望送り) | 凍結済みPDFエンジンの改修はリスク大。「紙は直通のみ・Webは乗換まで」と役割分担を資料に明記 |

## 8. 環境メモ

- Mac(macOS 12 Monterey・古い)。**Homebrewが不調**なので brew 前提の手順は避ける
  (これがr5py不採用の一因)。gh CLI は導入済み
- Python3 + pip。使用ライブラリ: pandas, numpy, shapely, folium, branca, requests, pytest
  (F7で segno を追加予定)。geopandas・Javaは不要のまま保つ
- PDF生成はChrome(ヘッドレス)に依存。`make_pair_timetable.py` 内の `CHROME` 定数がパス
- 作業ディレクトリは外付けSSD。macOSが作る `._*` ゴミファイルは gitignore 済み
- 実行はすべて**プロジェクトルートから** `python3 gap_map/スクリプト名.py` の形

## 9. 引き継ぎセッションの作法(開発者との取り決め)

1. マイルストーン1つ=1セッション。完成条件を満たしたら検証の実行結果を見せ、
   開発者が確認してからコミット→次へ

   **importして使うだけ。改変しない**(例外はF7のQR欄追加のみ)。

   `sorted()` で固定した(到着時刻は不変。冪等性要件のための最小修正。コミット 9b55f39)。
