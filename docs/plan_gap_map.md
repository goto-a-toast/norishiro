## 1. 目的とゴール

作るものは1つの「計算エンジン」と2つの「出口」。

```
 [GTFSフィード群]──┐
 [施設データ]      ├─→ 到達時間計算エンジン ──→ 出口A: 交通空白マップ(HTML地図)
 [人口メッシュ]  ──┘      (transit_core.py)   └→ 出口B: 個人向け経路検索(CLI)
```

- **出口A(メイン)**: 地域を500m四方のマス目(メッシュ)に区切り、
  各マスから病院・スーパーへ公共交通でどれだけ行きやすいかを色分け地図にする。
  行きにくいマスに人口が多ければ、そこが「交通空白地帯」。
- **出口B(再利用デモ)**: 同じエンジンで「○○バス停から△△まで、9時出発の経路」を答える。
  エンジンが1つであることが作品のアピールポイントになる。

### 再利用を成立させる設計上のルール(最重要)

エンジン(`transit_core.py`)には**メッシュ・施設・人口の概念を一切持ち込まない**。
エンジンが知っているのは「バス停」と「時刻」だけ。

- 入力: 「出発できるバス停と、そこに立てる時刻」の辞書(どこから歩いて来たかは呼び出し側の都合)
- 出力: 「全バス停への最早到着時刻」+「経路をたどるための記録」

空白マップは「メッシュ中心から徒歩圏のバス停」を入力にし、
経路検索は「指定された出発バス停」を入力にするだけの違いになる。
この境界を守れば、後でエンジンをr5py等に差し替えることも可能(§4)。

## 3. 段階的拡大(山形市+上山市 → 県全域)に耐える設計

やりがちな失敗は「山形市用のコード」を書いてしまい、拡大時に書き直しになること。
対策は**「対象範囲」をコードでなく設定ファイルに追い出す**こと。

```python
# gap_map/config.py(抜粋・このまま使ってよい)
TARGET_MUNICIPALITIES = ["山形市", "上山市"]   # 県全域にするときはここに市町村名を足すだけ
MESH_LEVEL = 4                                  # 4 = 500mメッシュ
TARGET_DATE = "20260610"                        # 分析する日(平日の水曜。フィード有効期間内で祝日でない日を選ぶ)
```

- **GTFSは31フィード全部を毎回読み込む**(合計しても数十MB・数十万行で、pandasで問題ない規模。
  実測: 最大の山交バスで1,434停留所・29,726時刻行)。
  「対象範囲」はメッシュ側(=どの市町村のメッシュを出発点にするか)だけで決まる。
  こうするとフィードの選別ロジック自体が不要になり、市をまたぐ路線も自然に扱える。
- 市町村の境界は国土数値情報N03(行政区域)のGeoJSONを使い、
  「メッシュ中心がその市町村ポリゴンの中にあるか」で対象メッシュを決める(shapelyを使用)。
- フィードごとにIDが重複しうるので、読み込み時に必ず `stop_id` と `trip_id` に
  フィード名の接頭辞を付ける(例: `山形交通:1234`)。**これを最初からやること。**

県全域への拡大 = `TARGET_MUNICIPALITIES` を35市町村にする + 人口メッシュCSVを数ファイル追加DL。
コード変更はゼロが目標。

## 4. エンジンの技術選定: r5py vs 自作RAPTOR

### r5pyとは

Conveyal社のR5という**Java製**経路探索エンジンをPythonから呼ぶライブラリ。
`travel_time_matrix()`(多数地点間の所要時間表)と `detailed_itineraries()`(経路詳細)があり、
機能面では今回の2つの出口を両方カバーする。学術・自治体分析での実績も多い。

### 比較

| 観点 | r5py | 自作RAPTOR |
|------|------|-----------|
| 導入 | JDK 21 + OSM道路データ(PBF)の切り出しが必要 | pip追加ほぼなし(shapely/folium程度) |
| 徒歩の正確さ | 道路網に沿う(正確) | 直線距離×迂回係数1.3(近似) |
| トラブル時 | Java側のエラーは初心者に解読困難 | 全部Pythonなので自分で追える |
| 検証 | ブラックボックス気味 | 完成済みのペア時刻表PDFと突き合わせできる |
| 作品性 | 「r5pyを使いました」 | 「エンジンを自作し2用途に使い回した」と言える |
| 規模 | 全国級もOK | 県全域までは余裕(下記見積り) |

### 結論: 今回は自作RAPTOR。r5pyは見送り(ただし逃げ道を確保)

決め手は3つ。

1. **導入障壁が3つ重なる**: Java(このMacはmacOS 12と古く、しかもHomebrewが不調のため
   Adoptiumの.pkgで手動インストールになる)+ OSM PBFの県域切り出し(osmium等の別ツール)+
   jpypeまわりのトラブルリスク。初心者環境では1つでも詰まると全体が止まる。
2. **規模的に不要**: RAPTORは単純なアルゴリズムで、県全域(推定1万停留所・数十万時刻行)でも
   1出発点あたり0.1秒未満、人口のある全メッシュ(数千)を回しても数分〜十数分で終わる見込み。
3. **検証資産がある**: 第1部で作った時刻表PDF(例: 山形駅前→県立中央病院 乗車約27分)と
   エンジンの答えを直接照合できる。

**再検討の条件**(こうなったらr5pyに乗り換える):
鉄道込みの複雑な乗り継ぎを厳密にやりたくなった/徒歩経路の近似が結果を歪めていると分かった/
他県展開などで規模が跳ねた。§1のルールを守っていれば、差し替えは
`compute_access.py` の「エンジンを呼ぶ1関数」の置き換えで済む。

### RAPTORとは(初心者向け説明)

「乗り換え回数0回で行ける所→1回で行ける所→2回で…」と輪を広げていく探索方法。

```
入力: 出発できるバス停の集合と、各バス停に立てる時刻
ラウンド1: いま立てるバス停から乗れる最も早い便を全路線で探し、
           その便が停まる先のバス停の「最早到着時刻」を更新する
        → さらに300m以内の別のバス停へ徒歩で移動した場合も更新する(乗換)
ラウンド2〜3: 同じことを繰り返す(=乗換1回、2回)
出力: 全バス停の最早到着時刻
```

実装のポイント(実装セッションへの指示に含めること):

- 時刻はすべて「0時からの分」の整数で扱う(`25:10:00` は 1510分。既存コードの `to_minutes` を流用)
- RAPTORでいう「路線」は route_id ではなく**停車パターン**
  (stop_idの並びが完全に同じ便のグループ)。trips を停車列でグループ化して作る
- 運行日の判定は既存の `build_daytype_groups` / `analyze_exceptions` の考え方を移植し、
  「指定日に走る service_id 集合」を calendar + calendar_dates から決める
- 各バス停に「どの便で・どこから来たか」を記録しておくと、あとで経路を逆にたどれる(出口Bで使用)
- 乗換の最小時間は3分+徒歩時間とする(config化)

## 5. パラメータ一覧(config.py に集約する)

高齢者の通院を想定した既定値。**すべて config.py に置き、コードに直書きしない。**

```python
# --- 徒歩モデル ---
WALK_SPEED_M_PER_MIN = 60     # 高齢者想定の歩行速度(60m/分 = 時速3.6km)
WALK_DETOUR = 1.3             # 直線距離→実際の道のりへの割増係数
MAX_WALK_TO_STOP_M = 800      # バス停まで歩ける上限(直線距離)
TRANSFER_WALK_M = 300         # 乗り換えで歩ける停留所間距離
MIN_TRANSFER_MIN = 3          # 乗り換えの最小余裕時間(分)

# --- 探索 ---
MAX_TRANSFERS = 2             # 乗り換え上限(=RAPTORのラウンド数3)
DEPART_TIMES = ["07:00", "07:30", ..., "10:30"]  # 30分刻みで試して最短を採用

# --- 通院シナリオ(指標②) ---
HOSPITAL_ARRIVE_BY = "11:00"  # 午前中にこの時刻までに病院に着けること
HOSPITAL_STAY_MIN = 90        # 受診にかかる滞在時間(分)
HOME_RETURN_BY = "17:00"      # この時刻までに帰宅できること

# --- 空白判定(指標のしきい値) ---
GAP_THRESHOLD_MIN = 60        # 片道これ以上かかるなら「行きにくい」
WALKABLE_FACILITY_MIN = 15    # 施設まで徒歩15分以内なら、バスがなくても空白としない
```

徒歩時間の計算式(全スクリプト共通のヘルパーにする):
`徒歩分 = ハバサイン距離(m) × WALK_DETOUR ÷ WALK_SPEED_M_PER_MIN`
(ハバサイン距離の関数は step5_overpass_shops.py に実装済み。コピーして使う)

### 6.1 メッシュ(標準地域メッシュ)

「標準地域メッシュ」は日本の統計で使われる公式のマス目で、
コード(数字列)と緯度経度が計算式だけで相互変換できる。**500m=4次メッシュ(コード9桁)を使う。**

`meshcode.py` に実装する変換式(この式をそのまま実装すればよい):

- 1次(4桁 `PPQQ`): 南端緯度 = PP ÷ 1.5、西端経度 = QQ + 100。高さ40分×幅1度
- 2次(+2桁 `rc`): 1次を縦横8等分。南端に r×(5/60)度、西端に c×(7.5/60)度を加算
- 3次=1km(+2桁 `rc`): 2次を縦横10等分。南端に r×(30/3600)度、西端に c×(45/3600)度を加算
- 4次=500m(+1桁 `n`): 3次を4等分。n=1南西, 2南東, 3北西, 4北東。
  緯度半分=15/3600度、経度半分=22.5/3600度
- 中心点 = 南西端 + マスの半分

検算用テストケース(必ず単体テストにする。数値は検算済み):
山形駅(北緯38.2484・東経140.3278)を含む3次メッシュは `57402296` で、
その南西端は 北緯38.241667度・東経140.325度になるはず。
逆変換(緯度経度→コード)も往復で一致すること。
※コードの桁の並びは「緯度方向の数字が先、経度方向が後」(2次の `22` は南から2番目・西から2番目)。

山形県はほぼ1次メッシュ **5740・5739・5839・5840** に収まる。
山形市+上山市の段階では **5740だけで足りる**。

### 6.2 施設データ → `data/facilities.csv`

異なる出どころのデータを、**先にこの共通形式へ変換してしまう**のが統合のコツ。
以降の計算コードは出どころを知らなくてよくなる。

| 列名 | 例 | 説明 |
|------|----|------|
| name | 山形県立中央病院 | 施設名 |
| category | hospital / supermarket | 施設種別(この2値から始める) |
| lat, lon | 38.28, 140.31 | 緯度経度(10進) |
| source | ksj_p04 / osm | 出どころ(検証用) |

- **病院**: 国土数値情報 P04(医療機関)。
  https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-P04-v3_0.html から山形県分をDL。
  GeoJSON形式があればそれを(pandas+json標準ライブラリで読める)、
  Shapefileしかなければ `pip install geopandas` で読む。
  属性「医療機関分類」で **病院(コード1)のみ** 採用(診療所は歯科等が混ざり過剰になるため第1弾では外す。config でON/OFFできる作りにする)。
- **スーパー**: OSM Overpass API。**step5_overpass_shops.py がほぼそのまま使える**
  (User-Agent対応済み)。`shop=supermarket` のみ採用(コンビニは「食料品アクセス」の
  論点が変わるので第1弾では外す。これも config で切替可能に)。
  検索範囲は市名指定でなく county 全域の bbox 指定に変える(拡大に備える)。
- 統合時の重複(同じ病院が両方にある等)は今回は気にしない(カテゴリが分かれているため実害なし)。

確認方法: 県立中央病院・大学病院と、ヤマザワ数店舗がCSVに入っていること。件数の目安は
病院10〜20件・スーパー30〜60件(山形市+上山市)。桁が違ったら抽出条件を疑う。

### 6.3 人口メッシュ → `data/target_meshes.csv`

- 入手(手作業でよい): e-Stat「統計地理情報システム」 https://www.e-stat.go.jp/gis
  → 統計データダウンロード → 国勢調査 → 2020年 → 「4次メッシュ(500mメッシュ)」
  → 「その1 人口等基本集計に関する事項」→ メッシュ番号 **M5740** のCSVをDL → `data/` に置く。
  ログイン不要。CSVには `KEY_CODE`(=メッシュコード)と人口総数の列がある
  (列名はファイル内のヘッダー2行目で確認すること。「総数」系の最初の列が人口総数)。
- `prepare_meshes.py` がやること:
  1. 人口CSVを読み、人口>0のメッシュだけ残す(山間部の無人メッシュを最初から捨てる)
  2. 各メッシュ中心を N03 行政区域GeoJSON(https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-N03-2024.html
     山形県分)と照合し、`TARGET_MUNICIPALITIES` 内のものだけ残す(shapelyの `contains` を使用)
  3. `target_meshes.csv`(meshcode, lat, lon, population, municipality)を書き出す

確認方法: 対象メッシュの人口合計が住民基本台帳の実勢
(山形市 約24万 + 上山市 約2.6万 ≒ **27万人前後**)に近いこと。
大きくズレたら市町村判定かメッシュ変換のバグ。

### 6.4 GTFS中間データ(build_network.py)

31フィードを読み、エンジンがすぐ使える形に前処理して保存(pickleで可)。

- 全フィードの stops / stop_times / trips / calendar / calendar_dates を接頭辞付きで結合
- `TARGET_DATE` に走る service_id を決定(既存ロジック移植)→ その日の trips だけ残す
- 停車パターン(stop_id列が同一のtrip群)にグループ化し、パターンごとに
  「停車順のstop_idリスト」と「各tripの時刻表(分)」を持たせる
- バス停間の徒歩乗換表(300m以内のペア)を事前計算
  ※全停留所総当たりでも1万×1万は重いので、緯度経度を0.005度くらいの格子に切って
  近傍格子だけ比較する(この工夫だけで数秒になる)

## 7. 「到達しやすさ」指標の定義

### 指標①: 最短所要時間(まずこれを完成させる)

> そのメッシュから **最寄りの病院/スーパーそれぞれ** へ、平日午前(7:00〜10:30の30分刻みの
> どこかに出発して)公共交通+徒歩で行くときの最短の所要時間(分)。

- 「所要時間」= 出発時刻から施設到着までの全部(徒歩+待ち+乗車+乗換待ち)
- 出発時刻を複数試すのは、「たまたま9:00発だとバスが行った直後」問題を避けるため
- 徒歩だけで着けるなら徒歩時間そのもの(バスに乗る必要はない)
- どうやっても着けない場合は「到達不能」という別カテゴリにする(999分等の偽値で埋めない)

### 指標②: 通院可能性(独自性のある指標。①の後に追加)

> 「午前11時までに病院に着き、90分滞在して、17時までに帰宅できるか」(Yes/No)
> +できる場合の「拘束時間」(家を出てから帰るまでの合計分)

- 行き: 指標①と同じ探索で、11:00までに着ける最も遅い到着を採用
- 帰り: 到着時刻+90分を出発時刻として、**病院側から自宅メッシュへ向けて同じエンジンをもう一度
  順方向に回す**だけ(逆方向探索は実装しない。これで十分)
- 通院はできるが拘束6時間、のような「数字上は空白でないが実質厳しい」地域が見える。
  これがこの作品の主張になる

### 空白判定(地図の強調表示用)

```
空白メッシュ = 人口 > 0
             かつ 病院まで徒歩15分では行けない
             かつ (指標①が60分超 または 到達不能 または 指標②がNo)
```

しきい値(60分・15分)は議論の余地があるので、**config変更→地図再生成が1コマンドで
できること**を要件にする(審査でも「しきい値を変えると?」は聞かれやすい)。

### 出力 `output/access_mesh.csv` の列

meshcode, population, municipality, nearest_stop_name, walk_to_stop_min,
time_to_hospital_min, hospital_name, time_to_super_min, super_name,
hospital_visit_ok (Yes/No), visit_total_min, is_gap (True/False)

## 8. 可視化(make_map.py)

Foliumで1枚のHTML(`output/gap_map.html`)を作る。構成:

1. **ベース地図**: OpenStreetMapタイル。中心=山形駅、ズーム12
2. **レイヤ1「病院への所要時間」**(既定ON): メッシュを塗り分けたGeoJSONレイヤ。
   メッシュのポリゴンは meshcode.py から自分で生成する(geopandas不要)。
   色分け: 〜15分 / 〜30分 / 〜45分 / 〜60分 / 60分超 / 到達不能 の6段階。
   連続的な黄→赤系(到達不能のみ灰色+斜線的な濃灰)。色だけに頼らず
   ツールチップに数値を出す(色覚多様性への配慮。審査観点でもある)
3. **レイヤ2「スーパーへの所要時間」**(既定OFF): 同上
4. **レイヤ3「通院可能性」**(既定OFF): Yes=薄緑 / No=赤 の2値
5. **レイヤ4「交通空白メッシュ」**(既定ON): is_gap=True のメッシュだけ赤枠で強調
6. **施設マーカー**: 病院=十字アイコン、スーパー=カゴアイコン。名前をポップアップ
7. **バス停レイヤ**(既定OFF): 小さい丸。多いので既定OFFにする
8. `folium.LayerControl()` で切替UI。各メッシュのツールチップに
   「人口/最寄り停留所/病院まで○分/通院可否」を表示

性能注意: メッシュ数千個のGeoJSONはHTMLが数MBになるが実用範囲。
県全域で重くなったら「人口>0のみ描画」(実装済みのはず)と簡略化を検討。

## 10. 既知の限界(README にも書く。正直さは作品の信頼性になる)

1. **徒歩は直線距離×1.3の近似**。川や線路で分断された場所は実際より甘く出る
2. **出発点はメッシュ中心**。500m四方の中の位置差(最大350m≒徒歩6分)は丸められる
3. **特定の1日(平日)のダイヤで計算**。冬ダイヤ・減便期は別途再計算が必要
4. デマンド交通・タクシー・鉄道駅の構内徒歩などは対象外
   (山形鉄道フィードは読み込むが、駅とバス停の乗換は300m徒歩ルールで近似)
5. OSMのスーパー網羅性は地域差がある(第1部の調査で山形市内は実用的な水準と確認済み)

## 11. 用語ミニ辞典

- **メッシュ / メッシュコード**: 全国を緯度経度で機械的に区切ったマス目と、その番号。
  番号から場所が計算で復元できるのが利点
- **RAPTOR**: 時刻表ベースの経路探索アルゴリズム。「乗換回数の輪」を広げていく方式
- **ハバサイン距離**: 緯度経度2点間の直線距離(球面上)。step5に実装済み
- **国土数値情報**: 国交省が配る地理データ集。P04=医療機関、N03=市町村境界
- **e-Stat**: 政府統計のポータル。国勢調査のメッシュ人口はここから
- **shapely**: 「点がポリゴンの中にあるか」等を計算するPythonライブラリ(pipで入る)
- **Folium**: Pythonから地図HTMLを作るライブラリ(中身はLeaflet.js)

## 13. M8: モビリティ・カルテ(出口C)の実装計画(2026-07-15 追加)

元ネタ: docs/idea_karte.md(別セッションの要件メモ)。
本章は分析凍結(M1〜M5・git tag `gap-map-analysis-v1`)の**あとに追記された生きた計画**。
凍結済みの計算結果を**読み替えるだけ**で、エンジン・分析値には一切触らない。

### 13.1 目的とストーリー

- 空白マップ(第2部)は「地域の診断」。これを「**個人の診断書**」に読み替える入口を1枚足す:
  住所を入れる → 「あなたの家の移動のようす」カードが出る →
  「この家からの時刻表はこちら」で第1部・かんたん/しっかりモードへつながる
- これにより「診断(カルテ)→処方箋(あなた専用の時刻表)」という一本のストーリーになり、
  第1部と第2部が「2作品の同居」に見える弱点を解消する
- 分析はもともと車なし前提なので、計算を変えずに「**免許返納シミュレーション**
  (車を手放したらこうなる)」という枠付けができる

### 13.2 決めごと(論点と結論)

| # | 論点 | 結論 | 理由 |
|---|---|---|---|
| 1 | ページの置き場所 | `webapp/karte/`(第3部の一員) | 静的サイト方式(§3.3)のまま。GitHub Pagesで動く |
| 2 | 位置の入力方法 | ①住所検索(国土地理院API)②GPS(実装済みの流用)③地区から選ぶ(保険) | 地図クリックは**オプション送り**(地図ライブラリ依存を増やさない) |
| 3 | 計算はどこで? | **すべてデータ工場(Python)で事前計算**。A〜E評価も平均比較も karte.json に焼き込む | 「Webは計算しない」原則。ブラウザがするのは「最寄りメッシュを距離で選ぶ」だけ(GPS地区判定と同じ前例) |
| 4 | プライバシー | 住所・座標を**保存しない・自サーバーに送らない**。外部送信は地理院APIへの住所文字列のみ | READMEに明記(審査観点)。静的サイトなので自然に満たせる |
| 5 | 文言 | 「診断書」「カルテ」は比喩。断定を避け「めやす」と書く。系統・メッシュ等の内部語は出さない | 医療的診断との誤解防止(idea_karte §4) |
| 6 | しきい値 | `config.py` に `KARTE_` 接頭辞で集約。空白判定(§7)と同じ値を参照する | 「しきい値を変えると?」に1コマンドで答えられるように |

### 13.3 設計(データの流れ)

```
output/access_mesh.csv(凍結・読むだけ)
data/mesh_districts.csv + mesh_subdistricts.csv(地区・サブ地区の対応)
webapp/data/districts.json(地区名)
        │
        ▼  gap_map/export_karte_data.py(新規。分析データのあるMac/Windowsで実行)
webapp/data/karte.json(新規・コミット対象。目安150KB)
        │
        ▼  webapp/karte/index.html + app.js + style.css(新規)
住所入力 → 地理院APIで緯度経度 → 一番近いメッシュを選ぶ → カードを表示
```

**karte.json の中身(1メッシュ1行)**: 緯度経度(5桁丸め)/最寄り停名・徒歩分/
病院・スーパーへの所要分と施設名/通院可否と拘束分/総合評価(A〜E)/地区ID。
先頭に「市平均・地区平均・しきい値・評価の説明文」をまとめた `meta` を持つ。
すべて access_mesh.csv の列(§7末尾)の読み替えで、**新規計算はゼロ**。

**A〜E評価の初期案**(export時に計算。configで変更可):
病院所要・通院可否・徒歩分の3点から機械的に割り当て、E=空白判定(§7)と一致させる。
採点表を出力して開発者が確認してから確定する(サブ地区分割と同じ進め方)。

**地理院の住所検索API**(初心者向けメモ): `https://msearch.gsi.go.jp/address-search/AddressSearch?q=住所`
に問い合わせると候補地点の緯度経度がJSONで返る。無料・キー不要・ブラウザから直接呼べる。
ただし外部サービスなので**M8-2の最初に疎通確認**を完成条件に入れる(仕様変更リスクは§13.6)。

### 13.6 既知の限界・リスク(READMEにも書く)

1. **出発点はメッシュ中心**(§10-2と同じ)。住所ピンポイントではなく「500m四方のめやす」。
