# 全国展開キット 設計計画書(R1〜R4)

他の市区町村でも「時刻表Web+空白マップ+カルテ」のデータ一式を作れるようにする。

- 作成日: 2026-07-19
- 発端: 開発者要望「他の市区町村のデータでも応用できるようにしたい。山形市、上山市以外の
  地域でもデータを作成できるようなUI(どういうことをやっているか分かる説明がついたもの)が欲しい」
- 開発者確認済みの決定(2026-07-19): **①対話式ウィザード方式**(ターミナルで質問に答える形。
  各工程に日本語の説明を付ける)**②学区データが無い地域向けの代替地区分けも用意**
- 対象読者: 実装セッションと開発者本人(プログラミング初心者・pandas経験あり)。
  発注の作法は docs/plan_gap_map.md §12 の定型文で「今回は R1」のように指定する
- READMEの公言「他の自治体もフォークして自分の市で再生成できます」を実体化する取り組み。
  応募資料でも「再現性・横展開」の章の実証になる

## 1. 全体方針(最重要ルール)

1. **山形版の動作は1バイトも変えない**。地域設定ファイル `data/region.json` が**無ければ**
   現行の山形設定がデフォルトとして働く。既存テスト・生成済みデータは完全互換。
   リグレッションは「山形設定での再生成が現データとバイト一致」で確認する
2. 凍結資産(transit_core.py・build_network.py)には触れない(汎用化は設定と入口だけで済む
   設計になっている: フィード接頭辞方式・エンジンは停留所と時刻しか知らない)
3. 人手で直すもの(事業者の表示名・電話・地区名かな等)は、districts_master.csv と同じ
   「自動初期値+人間の編集列+再実行でも編集を消さない」流儀のマスタCSVにする

## 2. 調査済みの山形固有部分(汎用化の対象)

| 場所 | 内容 |
|---|---|
| gap_map/config.py | TARGET_MUNICIPALITIES・GTFS_FEED_DIRS・POP_MESH_FILES・N03/P04パス・TARGET_DATE |
| gap_map/export_web_data.py | 基準フィードassert(山交)・DEMAND_PHONE・OPERATOR_INFOの直書き辞書 |
| gap_map/make_districts.py | CITY_CODE直書き・「山形市=A27ポリゴン/上山市=P29最寄り校」の市名分岐 |
| gap_map/make_destinations.py | CORE_HOSPITALS・まちなかスポット直書き |
| gap_map/download_gtfs.py | TARGET_OPERATORS直書き(yamagata_gtfs_feeds.csv参照) |
| 各検算 | 総人口276,482人・空白15,418人・817メッシュ・PDF照合など山形の確定値に依存 |
| webapp | かんたん画面1の市タブ直書き(index.html)、shikkari/karte app.js の `["山形市","上山市"]` ループ、karteの「山形市の平均より」文言 |

meta.json の demand_phone / operators は既にデータ供給側なので、元の辞書をマスタCSV化
すれば Web 側の流れは既存のまま変わらない。

## 3. マイルストーン

### R1: 設定の外出し(互換レイヤ)

- 新規 `gap_map/region.py`: `data/region.json` が**あれば**読み、**無ければ山形デフォルト**を
  返す読み込みヘルパ。config.py はこれを経由する
- `export_web_data.py` の OPERATOR_INFO / DEMAND_PHONE を
  `data/operators_master.csv`・`data/demand_phone_master.csv` に外出し
  (山形の現値で初期生成してコミット=山形は無変更)
- `make_destinations.py` の CORE_HOSPITALS・まちなかスポット → region設定へ
- **検算の切替**: region.json に期待値(山形: 総人口276,482・空白15,418・817メッシュ等)を
  持ち、期待値が無い地域では「厳密一致チェック」→「観測値の表示+妥当性チェック
  (0でない・重複なし等)」に自動で切り替える(★NGで止まらず、人間が見て判断できる
  採点表を出す=これまでの進め方と同じ)
- 完成条件: 山形設定で全pytest緑+export_web_data再生成が現データと**バイト一致**
  (再生成はMac)。単体テスト追加(region.jsonあり/なしの分岐)
- 確認方法: pytest結果+Macでの再生成差分ゼロの確認

### R2: 地区分けの汎用化(代替方式)

- `make_districts.py`: 市町村ごとに方式を region.json で選択:
  - `a27_polygon`(学区ポリゴン。現在の山形市方式)
  - `p29_nearest_school`(最寄り小学校で近似。現在の上山市方式)
  - `municipality`(**代替**: 市町村ぜんたいで1地区。学区データが無い地域用)
- CITY_CODE(市町村コード)は N03(行政区域)の属性から自動取得(直書き廃止)
- `municipality` 方式で広すぎる地区は、既存の **make_subdistricts.py(人口クラスタ自動分割・
  実装/テスト済み)をそのまま再利用**して分割する。地区名の人手修正も既存
  subdistricts_master.csv の仕組みがそのまま使える(新規実装ほぼゼロで代替が成立)
- 完成条件: 山形設定の出力不変+手作りフィクスチャで3方式の単体テスト
- 確認方法: pytest結果

### R3: 対話式ウィザード `gap_map/setup_region.py`(本丸)

- `python3 gap_map/setup_region.py` で起動。日本語で順に質問:
  1. 都道府県・対象市町村名(複数可)
  2. GTFSフィード: gtfs-data.jp での探し方を案内し、ダウンロード済みフォルダの場所を
     確認(存在チェック)
  3. 地区分け方式(市町村ごとに。A27の収録状況の確かめ方も案内)
  4. 分析日(フィード有効期間内の平日・非祝日の選び方を説明)
  5. 事業者の表示名・案内電話(わからなければ空でOK。あとでマスタCSVを直せると案内)
- **各ステップに「今なにを設定しているか」「なぜ必要か」の説明文を必ず表示**(要望の核)。
  手動ダウンロードが必要なもの(N03・国勢調査人口メッシュ・P04)は、対象地域から
  **必要なファイル名(1次メッシュ番号は座標から計算)と入手URL・置き場所を具体的に**案内し、
  置いたら存在チェックする
- 最後に `data/region.json` を書き出し、実行すべきコマンド一覧を順序どおり表示。
  `--run` を付けると各工程を「説明→実行→検算表示→続行確認」で連続実行
- 完成条件: **天童市**(GTFS・N03・人口メッシュ・P04がすべて手元に揃っており追加DL不要)を
  対象にウィザードを通し、天童市版の webapp/data 一式が生成できること(実地はMacで開発者)
- 確認方法: 入力をモックした単体テスト+天童市試走の画面貼り付け

### R4: webappの動的化+フォークガイド

- かんたんモード画面1の市タブを districts.json の municipality 一覧から**JSで生成**
  (HTML直書き廃止。1市だけならタブ自体を出さない)。shikkari/karte の市ループも動的化
- karteの比較文言「山形市の平均より〜」→「この地域の平均より〜」
- 玄関ページ(index.html)の作品説明・確定数値は作品固有なので触らず、フォークガイドに
  「差し替える場所リスト」として明記
- 新規 `docs/guide_fork.md`(初心者向け・日本語): 必要なもの→ウィザード→人手マスタ記入→
  GitHub Pages公開まで。READMEに「他の地域で使うには」節を追加してリンク
- 完成条件: 山形データでの表示が現状と完全同一(Playwright既存スイートでリグレッション)+
  疑似1市フィクスチャで「タブ非表示」等の新挙動を確認
- 確認方法: Playwright結果+スクリーンショット

## 4. 検証まとめ

- pytest全件(新テスト含む)/山形設定でのexport再生成バイト一致(Mac)/
  Playwright既存スイート(かんたん・しっかり・カルテ・あしたの時刻表)+R4の動的タブ分
- 実地検証: 開発者がMacで**天童市**を対象にR3ウィザードを試走→生成物をブラウザ確認
  (コミットはしない・検証のみ。検証後は region.json を消せば山形版に戻る)

## 5. 環境分担と順序

- コード・テスト・ドキュメント: 実装セッション(サンドボックス)
- 再生成・ウィザード試走(天童市): 開発者のMac
- 順序: R1 → R2 → R3 → R4(各マイルストーンごとにPR)
