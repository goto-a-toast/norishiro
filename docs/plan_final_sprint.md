## 1. 全体像: 「データ供給工場」と「3つの顔」

```
【Python側 = データ供給工場(手元で実行)】
 9フィードGTFS ─→ transit_core(凍結済みRAPTOR) ─→ export_web_data.py
                                                        │ 事前計算JSON一式
                                                        ▼
【Web側 = 静的サイト(GitHub Pages。サーバー計算なし)】
 webapp/
 ├─ index.html …… 玄関(3つの顔への入口)
 ├─ かんたんモード …… 高齢者本人・窓口支援者向け(3タップで時刻表)
 ├─ しっかりモード …… 家族・支援者向け(詳細表示+ブラウザ印刷)
 ├─ gap_map.html  …… 空白マップ(完成済みをコピー配置+説明ページ)
 └─ data/*.json   …… 事前計算した時刻表データ
```

設計原則(実装セッションはこれを守ること):

1. **Webアプリは計算しない**。JSONを読んで表示するだけ。今日の日付→ダイヤ種別の
   判定すら、Python側で作った対応表(§6のmeta.json)を引くだけにする
2. **素のHTML/CSS/JS**。フレームワーク・ビルドツール・CDN読み込みなし。
   `webapp/` をそのままアップロードすれば動く状態を保つ
3. デザイン言語は第1部PDFの紙面を継承する(ごぜん/ひる/ごご/よるの段、
   行き=青・帰り=橙、BIZ UDPゴシック)。「紙とWebで同じ見た目」が作品の一貫性になる

## 2. 論点1の決定: 事前計算の範囲

組み合わせ数の構造 = 出発「地区」数 × 行き先「施設」数。
実測値(凍結資産)から概算した比較:

| 案 | 地区数(目安) | ペア数 | バッチ時間(目安) | JSON容量 | 人手の追加作業 |
|---|---|---|---|---|---|
| A: 山形市+上山市 | 約43(小学校区) | 約950 | **15分前後** | 約6MB(1地区150KB) | 行き先の厳選のみ |
| B: 隣接込み9市町 | 約120 | 約2,600 | 40分前後 | 約18MB | 行き先厳選×9市町、学区データ確認×9 |
| C: 県全域 | 約230 | 約5,000超 | 1〜2時間 | 30MB超 | 未DLの22フィード追加+全35市町村の厳選 |

**推奨: 案A(山形市+上山市)。ただしバス路線網は凍結済みの9フィードのまま使う。**

- 「出発地区は2市、計算に使う路線網は9フィード」なので、市境をまたぐ便
  (山形市→天童の病院など)も正しく出る。案Aは「狭い」のではなく「出発点を絞る」だけ
- ボトルネックは計算時間ではなく**人手の品質作業**(行き先の厳選、地区名の妥当性確認、
  スポットチェック)。これは地区数に比例するので、締切前に増やすべきでない
- 案Cは品質面でも不利: 内陸〜庄内はJR抜き(READMEの限界6)で非現実的な結果になり、
  審査で突っ込まれる弱点を自分から作ることになる
- 拡大は「地区マスタと行き先マスタに行を足して工場を再実行するだけ」の設計にする(§6)。
  応募資料では「県全域はデータ追加のみで対応可能」と展望に書く

## 3. 論点2の決定: 「地区」の定義

| 候補 | 数(2市) | 高齢者に通じるか | データ入手 | 判定 |
|---|---|---|---|---|
| **小学校区** | **約40** | **◎ 「◯◯小学校の学区」は地域の共通語** | **国土数値情報 A27(ポリゴン)** | **採用** |
| 公民館区・地域振興区 | 約30 | ◎ | ✗ オープンデータなし(市サイトの表のみ) | 見送り |
| 大字・町丁目 | 数百 | △ 中心部は細かすぎ | e-Stat境界データ | 細かすぎ |
| 500mメッシュ代表点 | 817 | ✗ 「メッシュ574022964」は人に通じない | 手元にあり | 内部処理専用 |

- **地区 = 国土数値情報 A27(小学校区)の山形県分**。学区は統廃合が少なく、
  多少古い年度版でも実用上問題ない(最新年度版をDL。F0の人間作業)
- **地区の代表点 = その学区内で人口が最大の500mメッシュの中心**
  (`data/target_meshes.csv` から機械的に決まる。「その地区で一番人が住んでいる
  あたりから計算した」と一言で説明できる)
- 学区名がそのまま高齢者に通じない場合(「第◯小」等)に備え、地区マスタに
  「表示名」列を持たせ、人間が最終確認・修正できるようにする(F1)

## 6. データ設計(工場の出荷物 = webapp/data/)

**この節が実装の中心仕様。** 生成は `gap_map/export_web_data.py`(新規)が行う。
凍結資産(transit_core・build_network・第1部の運行日ロジック)は**importして使うだけ**で改変しない。

### meta.json(1個)

```json
{
  "generated": "2026-07-12",
  "valid_until": "2026-09-30",
  "day_types": {"weekday": "平日", "saturday": "土曜", "sunday_holiday": "日曜・祝日"},
  "date_table": {"2026-07-13": "weekday", "2026-07-18": "saturday", "...": "有効期間の全日分"},
  "demand_phone": [
    {"area": "山形市", "name": "(便名を人間が記入)", "tel": "(F0で人間が調査)"},
    {"area": "上山市", "name": "…", "tel": "…"}
  ]
}
```

- `date_table` は有効期間の全日について、凍結済みの運行日ロジック
  (calendar+calendar_dates → その日のダイヤ種別)をPython側で回して焼き込む。
  **祝日・お盆の特殊ダイヤ判定がJS側に一切不要になる**のがこの設計の肝

### districts.json(1個)

```json
[{"id": "d01", "name": "金井地区", "kana": "かない", "municipality": "山形市",
  "lat": 38.28, "lon": 140.32, "source_school": "金井小学校区"}]
```

### destinations.json(1個)

```json
[{"id": "f_kenchu", "name": "県立中央病院", "kana": "けんりつちゅうおう",
  "category": "hospital", "lat": 38.30, "lon": 140.31}]
```

- カテゴリは hospital / supermarket / town(まちなか)の3値
- **凍結済みの facilities.csv 153件から自動で載せない。** 厳選リスト(1市あたり
  病院5〜8・スーパー5〜8・まちなか3〜5、合計30〜45件)をCSVで人間が確定する(F2)

### timetables/{地区ID}.json(地区ごとに1個 ≒ 43個)

```json
{"district": "d01",
 "to": {"f_kenchu": {
   "board_walk_min": 6,
   "outbound": {"weekday": [
     {"dep": "07:22", "arr": "07:49", "board": "金井公民館前", "platform": null,
      "alight": "県立中央病院", "route": "N52", "ride_min": 27, "transfer": null},
     {"dep": "10:15", "arr": "11:02", "board": "金井公民館前", "alight": "県立中央病院",
      "route": "C2", "ride_min": 40,
      "transfer": {"at": "山形駅前", "wait_min": 8, "route2": "T20"}}
   ], "saturday": ["…"], "sunday_holiday": ["…"]},
   "inbound": {"…帰り方向、同じ形…"}
 }}}
```

生成方法(export_web_data.py の中身):

- 各(地区代表点×施設×ダイヤ種別)について、始発〜終バスの各出発候補で
  凍結RAPTORを回し、「異なる第1便」ごとに1行にまとめる(=時刻表になる)
- 直通が存在する時間帯は直通を優先。乗換1回は直通が無い時間帯の補完として載せる
- 徒歩30分(config値)以内にバスで行けない施設は `"unreachable": true` を出す
- 出力前にJSONの合計サイズを表示する。**合計10MB・1地区250KBを超えたら設計に戻る**

### 検算(F3の完成条件に使う)

1. 「山形駅前を含む地区 → 県立中央病院」の平日欄が、第1部PDF
   (`timetable_山形交通_山形駅前_県立中央病院.pdf`)の便と一致すること
2. date_table で 2026-07-20(海の日・月曜)が sunday_holiday、
   2026-08-13〜15 が sunday_holiday になっていること(第2部で確立した検証値)
3. ランダムな3地区でNAVITIME照合(±10分は徒歩モデル差として許容)

## 8. 論点5の決定: 応募資料の構成

### 作品概要文の骨子(この順で書く)

1. **課題を数字で**: 山形市・上山市で公共交通の空白地帯に15,418人(5.6%)。
   その高齢化率は43.2%と、2市平均30.3%より12.9ポイント高い。
   さらに「時刻表上は行けるのに通院が成立しない」隠れ空白に571人・高齢化率51%
   ——**移動に困る場所ほど、移動が困難な人が住んでいる**
2. **解決**: ①高齢者本人が3タップで使える時刻表Web+②窓口で配れる大活字PDF+
   ③政策判断のための空白マップ、を**1つの自作計算エンジン**から供給
3. **技術**: GTFS9フィード+国土数値情報+e-Stat。静的サイトなので運用費ゼロ、
   自治体がフォークして自分の市で再生成できる
4. **検証**: 実在時刻表・NAVITIME・現地データで照合済み(検証過程も公開)
5. **展望**: 県全域はデータ追加のみ/乗換経路の紙面化/JR等フィード公開への提言

## 9. リスクと逃げ道

| リスク | 逃げ道 |
|---|---|
| 締切が想定より近い | P1(F5・F7)を落としてもP0だけで作品は成立する構成にしてある |
| GitHub Pages がうまくいかない | 静的ファイルなのでVercel(Web画面からのアップロード)でも可。どちらもbrew不要 |
| SpeechSynthesis が端末で動かない | ボタンを自動非表示(本体機能に影響なし) |
| BIZ UDPゴシックTTF(計4MB)で初回表示が重い | まずシステムフォント表示→フォントは後読み(font-display: swap)。サブセット化は展望 |
| 学区名が住民感覚と合わない | districts.json の表示名を人間が上書きできる設計(F1)で吸収 |
| 印刷CSSのブラウザ差 | 検証はChrome基準と資料に明記(第1部PDFという確実な代替手段が既にある) |
