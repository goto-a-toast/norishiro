## 0. 要求(2026-07-07 開発者)

> 乗るバス停、おりるバス停を選択できるようにできませんか?
> (GPSが利用可能な環境であれば、現在地から近いのいくつかの場所から選択できるように)

対応の内訳:

| 要求 | 状態 |
|---|---|
| GPSで近い**地区**の候補から選ぶ | **実装済み**(コミット 09aad0b。両モードの📍ボタン) |
| **乗るバス停**を地区内の停留所から選ぶ | 本検討書の対象(データ再設計が必要) |
| **おりるバス停**を選ぶ | **非推奨と回答済み**: エンジンが施設徒歩圏800mの全停留所から「バス到着+徒歩の合計が最短」の停を便ごとに選択済み。選ばせると合計で遅い選択肢を提示することになる |

## 1. なぜ現状ではできないか(復習)

- Webアプリは計算しない(事前計算JSONの表示のみ)が根本設計
- 乗車側の事前計算は「地区代表点の最寄り停留所+150m以内(=同じ場所の別のりば)」に
  意図的に絞っている(export_web_data.py の SAME_PLACE_M)。徒歩圏800m全体を候補に
  すると、実測で1地区19停留所・1日3,567便が候補になり出力が破綻したため
- したがって「別の停留所から乗る」時刻表は**存在しない**。作るには停留所ごとの
  事前計算が必要

## 2. 設計案

### 案A: 停留所単位の事前計算+オンデマンド読み込み(フル版)

- 対象停留所: 地区の全メッシュ(住民がいる250mメッシュ)から徒歩圏
  (config.MAX_WALK_TO_STOP_M=800m)内の全停留所
- 出荷物(追加分):
  - `webapp/data/stops/{地区ID}.json` … 停留所索引(表示名・かな・lat/lon・
    のりば統合キー)。GPSの近傍候補と選択UIに使う
  - `webapp/data/timetables_stop/{地区ID}/{停留所キー}.json` … その停留所発の
    地区→38施設の時刻表(構造は現行の地区JSONと同じ)
- 読み込みは選択時のみ(初期表示は現行どおり地区代表)→ 実行時の重さは増えない。
  増えるのは**リポジトリとGitHub Pagesの容量**と**Macでの生成時間**
- URL設計: 現行 `#dNN/fNN` は不変(QR=F7も不変)。停留所指定は `#dNN/sKK/fNN`
- UX: かんたんモードの3タップは死守(画面3のカード下に小さく
  「べつのバス停から のる」を置く程度)。本体はしっかりモードの停留所ピッカー+
  GPS近傍停留所の候補表示

### 案B: 控えめ版(地区あたり主要2〜3停留所に限定)

案Aと同じ仕組みで対象停留所を便数上位2〜3に限定。データ2〜3倍で収まるが、
「家の近くの停留所が入っていない」が起こり、要求を部分的にしか満たさない。

### 案C: ブラウザで経路計算(否定)

「Webは計算しない」根本設計に反する。検算不能な計算結果を配ることになるため採らない。

## 3. 未知数と実測方法(着工判断の前に必ず実行)

サイズ・時間の見積りは停留所数に支配されるが、**この作業環境にはGTFSが無く実測
できなかった**。Macで以下を実行すると判断材料が出る(読み取り専用・出力変更なし):

```
python3 gap_map/estimate_stop_level.py
```

出力: 地区ごとの徒歩圏内停留所数(のりば統合後)/追加ファイル数/
現行地区JSONサイズからの外挿による総量見積り/生成時間の概算。

### 着工の判断基準(推奨)

| 指標 | 進めてよい目安 | 超えたら |
|---|---|---|
| 追加データ総量(見積り) | ≤ 150MB | 案B(主要2〜3停)に縮小 |
| 生成時間(Mac) | ≤ 2時間 | 同上 |
| 検算 | 「代表点の最寄り停」を選んだ結果が現行の地区JSONと一致すること | 原因究明まで着工しない |

GitHub Pagesの制限(サイト1GB・1ファイル100MB)には収まる見込みだが、
**gitの履歴は再生成のたびに肥大する**。着工時は「停留所データは再生成で全洗い替えに
なる」前提でコミット粒度を決めること(必要なら Git LFS も検討)。
