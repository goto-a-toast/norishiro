## 0. なぜこの文書があるか: プロトタイプで見つかった問題

F3のJSONをそのまま画面に出したところ、**計算機の内部語がそのまま利用者に露出した**。

| 実際に画面に出た表示 | 何が問題か |
|---|---|
| `Ｚ８０・Ｃ６`(系統記号がそのまま) | 高齢者には暗号。全角英数で機械的な印象。しかもこれは「路線ID」であって、バス停に立った人が目で確かめられる情報ではない |
| `ほかに50通りのルートがあります` | 計算結果の露出。50通りを見る手段も選ぶ理由も利用者にはない。不安にさせるだけ |
| `「山形市役所前」でのりかえ(Ｚ８０・Ｃ６)` | 乗換先の説明が系統記号。乗換地点で「どのバスに乗ればいいか」が分からない |

**F4の最重要課題は、この「翻訳」である。** 原則はひとつ:

> 画面に出してよいのは「バス停に立った利用者が、自分の目と耳で確かめられる情報」だけ。
> 計算機の都合(路線ID・経路数・内部データの列名)は一切出さない。

## 1. 翻訳ルール(R1〜R8。実装はこの章を仕様とする)

### R1: 主役は「行き先」(バスの正面表示)

- 各便の表示の中心は GTFSの `trip_headsign` = **バスの前面方向幕に実際に出ている行き先**。
  「10:20 → **『県庁前行き』にのる**」の形式を基本にする
- 表記は `『{headsign}行き』`。headsignが既に「行き/ゆき」で終わる場合は重ねない
- **データ実態(2026-07-06調査済み)**: 9フィード全てに trip_headsign 列があり
  **記入率100%**。したがって代替生成は「保険」だが、将来のフィード更新に備えて
  次のフォールバックを工場(export_web_data.py)に実装する:
  1. `trip_headsign`(空白除去後に非空ならこれ)
  2. その便の**終点停留所名**+「方面」(停車パターンの最後のstop名。全フィードで必ず作れる)
  - フォールバックが発動したら工場のログに件数を出す(品質低下の検知)

### R2: 系統番号は脇役(確認情報)

- 位置づけは「これから来るバスが正しいか、番号でも確かめられます」という**照合用の添え物**。
  小さい文字で `(かくにん) バスの番号: D56・C4` の形式
- **正規化**: `unicodedata.normalize("NFKC", …)` で全角→半角(`Ｄ５６・Ｃ４`→`D56・C4`)。
  この判定・変換は第1部 `make_pair_timetable.py` の `format_route_line()` に実装済みの
  考え方をそのまま移植する(「・」で分割した各部が `[A-Za-z0-9]{1,4}` なら系統コード)
- **系統コードらしくない路線名は出さない**。実データでは市営バス系が
  「上山市市営バス 市内循環線」のような説明文で、headsignと重複するだけなので非表示。
  例外として「循環」を含む場合のみ `市内循環線` の部分を出してよい(向きの確認に有用)
- **列挙上限3つ**: 時刻表ヘッダーに使える系統をまとめて出す場合は
  `バスの番号: N52 / C2 / D56 など` の形式で、3つを超えたら「など」で打ち切る
- 音声読み上げには系統コードを**含めない**(R8)

### R3: alt_routes(他ルート)は全面非表示

- かんたんモードでは**代表1本のみ**。「ほかに◯通り」系の文言は**全面禁止**
- 理由(例外を設けない根拠): F3の `collapse_transfer_alternatives()` は
  「出発時刻が20分以内に固まった乗換便グループから最速到着の1本」を代表にする。
  つまり折りたたまれた便は**代表とほぼ同時刻に出て、着くのは同じか遅い**便であり、
  利用者がそれを選ぶ合理的な場面が構造上存在しない。
  「次の便が3時間後」という問題は代表便どうしの間隔の問題であって、
  折りたたまれた代替便を見せても解決しない
- `alt_routes` フィールド自体はJSONに残す(F5しっかりモードで
  「くわしい人向け」に出す余地を残す)。かんたんモードのJSがこのキーを
  読むこと自体を禁止する

### R4: 乗換は「どこで降りて、次に何行きに乗るか」

- 表示は最大2行、すべて固有名詞は停留所の**表示名**のみ(IDや系統記号は使わない):

  ```
  「山形市役所前」で おりて、
  『県立中央病院行き』に のりかえ(19分 まち)
  ```

- データ側に `transfer.headsign2`(乗換後の便のheadsign)を追加する(§2)
- `wait_min` は「◯分 まち」として必ず出す(待ち時間の長さは乗る/乗らないの
  判断材料であり、隠すとかえって不信になる)
- 一覧上は「のりかえ1回」バッジ(橙)のみ。詳細2行は「のりかたカード」(§3)に出す

### R5: 時刻の表記(第1部PDFの紙面ルールを踏襲)

- 段組: **ごぜん**(〜10:59)/**ひる**(11:00〜12:59)/**ごご**(13:00〜17:59)/
  **よる**(18:00〜)。12時間表記(ごご1:41)。この区分・理由は
  `make_pair_timetable.py` の `to_dan()` のコメントどおり(「ごぜん11時/ごご0時」の
  迷い回避、「ごご7時」と「ごぜん7時」の読み間違い防止)
- 24:00以降の深夜便(GTFSの25:10表記)が現れた場合は「よる」の段に
  `深夜1:10` と表示する(現データにはalmost無いが、JSのヘルパーに含めておく)
- 「つぎのバス」の時刻だけは例外的に24時間表記も併記しない。`ごご2:41` に統一
  (時計カードには「あと◯分」があるので誤読の実害がない)

### R6: のりば

- `platform` があるときだけ `3番のりば` バッジを出す。無いときは何も出さない
  (第1部と同じ「一意に決まる場合のみ表示」の原則。データ側で担保済み)

### R7: ダイヤの種別と有効期限

- `※きょうは「平日」ダイヤです(自動判定)` を画面3に常時表示
  (meta.jsonの date_table を引くだけ。判定ロジックをJSに書かない)
- 有効期間外の日は時刻表を出したまま
  `※きょうはこの時刻表の対象外の日です。市の窓口にお問い合わせください` を優先表示
- `この時刻表は 2026年9月30日 まで有効です` を末尾に

### R8: 音声読み上げも「翻訳後の文」だけを読む

- 読み上げ文の組み立てに使ってよいのは: 時刻(「ごご2時41分」形式)・あと何分・
  バス停名・headsign・歩く分数・のりかえ案内。**系統コードとのりば番号は読まない**
  (「ディーごじゅうろく・シーよん」は音声では意味を成さない)
- 例: 「つぎのバスは、ごぜん10時20分、あと23分です。金井公民館前バス停から、
  県立中央病院行きに、のってください。バス停までは、あるいて約6分です。」

### 禁止語彙(レビュー用チェックリスト)

画面・音声に以下が出たらバグとして扱う:
`ルート` `系統` `route` `trip` `パターン` `メッシュ` `ID` 全角英数字の記号列、
`◯通り`、内部キー名(weekday等)の生表示、`到達不能`(→「バスでは行けません」)

## 2. データ側の変更(F3成果物への追加。マイルストーンF4-1)

**凍結資産(transit_core.py / build_network.py)には触れない。**
`Leg.trip_id` は「フィード名:trip_id」の接頭辞付きなので、
export_web_data.py(F3で作った非凍結コード)が各フィードの trips.txt を
自分で読み、trip_id→trip_headsign の対応表を作れば足りる。

### timetables/{地区ID}.json の便レコードに追加する項目

```json
{
 "dep": "10:20", "arr": "10:47",
 "board": "金井公民館前", "platform": "3",
 "alight": "県立中央病院",
 "headsign": "県立中央病院",          ← 追加(R1。フォールバック込みで必ず非空)
 "route": "N52",                      ← 変更(NFKC半角に正規化して出力。R2)
 "ride_min": 27,
 "transfer": {
   "at": "山形市役所前", "wait_min": 19,
   "headsign2": "県立中央病院",       ← 追加(R4)
   "route2": "C2"                     ← 変更(同上)
 },
 "alt_routes": 2                      ← 現状維持(かんたんモードは読まない。F5用)
}
```

- 正規化は**出力時**に行う(networkの中身は書き換えない)
- 系統コード判定(R2)はPython側に置かない: routeは正規化した文字列を
  そのまま出し、「コードらしいか」の表示判定はJS側の1関数
  `routeCodeOf(route)` に集約する(表示ルールはUIの責務のため)

### 完成条件(F4-1)

1. 全41地区JSONを再生成し、**全便レコードで headsign が非空**
   (フォールバック発動件数もログで0件のはず。0でなければ内訳を報告)
2. 乗換ありの全レコードに headsign2 がある
3. route / route2 に全角英数字が残っていない(`[Ａ-Ｚａ-ｚ０-９]` をgrepして0件)
4. 既存検算の再確認: 第二地区(d23)→県立中央病院の平日欄が第1部PDFと一致/

5. 合計サイズの増分を報告(headsign追加で+10〜15%見込み。30MBを超えたら相談)

設計の要点:

1. **のりかたカードと時計モードを一体化する**。「つぎの便」の情報が
   ①歩く→②乗る→③降りる の3ステップで1枚にまとまる。
   これがR1〜R6の翻訳ルールの表示先になる
2. **時刻チップは選択式**: 段組の時刻はボタン(最小60×60px)。タップすると
   のりかたカードがその便の内容に切り替わる(「あと◯分」は「つぎの便」を
   選んでいる時だけ表示)。初期選択=つぎの便。これにより
   便ごとに違う情報(headsign・乗換・のりば)を一覧に詰め込まずに済み、
   段組は紙面と同じ「時刻だけの密度」を保てる
3. **headsignの一括表記**: その方向・ダイヤ種別の全便が同じheadsignなら、
   段組の上に一度だけ「どの時刻も『◯◯行き』にのります」と書き、
   チップは時刻のみにする。headsignが混在する場合のみ、チップの下に
   小さくheadsign(6文字で省略)を添える
4. **乗換便のチップには橙の「※」印**。選択するとのりかたカードが
   ①〜③の間にR4の乗換2行を挟んだ4〜5ステップ表示になる
5. 乗換の詳しい説明・別のりば一覧など、カードに収まらない情報は
   **F5しっかりモードに送る**(かんたんモードには足さない)
6. 最終便後の時計モード: `本日の便は おわりました / あしたの始発は ごぜん6:50です`
   (翌日のダイヤ種別も date_table を引くだけ。現行実装済み・維持)
7. **電話番号欄**(新設): meta.json の demand_phone を地区名で引き、
   該当地区ならデマンド交通の予約電話を、それ以外は市の相談窓口を1行表示。
   `tel:` リンクにする(スマホならタップで発信)

### 文字サイズの基準(style.cssの変数にする)

| 要素 | サイズ | 備考 |
|---|---|---|
| つぎのバスの時刻 | 40px太字 | 画面3の最重要情報 |
| のりかたカードの本文・バス停名 | 28px | |
| 『◯◯行き』 | 28px太字・青 | R1の主役 |

### アクセシビリティ(実装時に守る)

- 市タブ・カテゴリタブは `role="tab"` + `aria-selected`(現行実装あり・維持)
- 時刻チップ選択は `aria-pressed`、のりかたカードは `aria-live="polite"`
  (選択変更を読み上げ環境に通知)
- 色だけに頼らない: 行き=青は「■ 行き」の文字併記、乗換=橙点は「※」記号併記
- フォーカス可視化(`:focus-visible` の太枠)。キーボードのみで3画面を往復できる

## 5. リスクと逃げ道

| リスク | 逃げ道 |
|---|---|
| 地区JSONが最大1.5MB(F3実測)で初回表示が重い | F4では対処しない(表示は地区選択後で、体感への影響は限定的)。F8の公開時に実測し、遅ければ「時刻表データを読み込んでいます…」表示+gzip配信(GitHub Pagesは自動)で吸収。それでも重ければ便レコードのキー名短縮を検討 |
| headsign記入率が将来のフィード更新で下がる | 終点名フォールバック(§2)が自動で埋める+工場ログで検知 |
| headsign混在で段組が煩雑になる地区がある | 一括表記の判定はダイヤ種別×方向ごと。混在時のチップ添え字は6文字省略。それでも読みにくいペアが見つかったら、その実例を持って相談(勝手に非表示にしない) |
| 時刻チップ選択式が高齢者に伝わらない懸念 | 初期選択=つぎの便なので、**タップしなくても従来ワイヤーと同じ情報が出ている**。チップ選択は「できる人向けの加点」であり、F4-4の第三者テストで詰まったら説明文言を調整する |
| SpeechSynthesisの声質・読み(「1:41」の読み方)が端末依存 | 読み上げ文は「ごご1時41分」のかな漢字で組み立てる(数字+記号を読ませない)。それでも不自然なら音声はF9の展望に降格可 |
